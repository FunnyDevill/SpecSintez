Документация по тестам
Общие сведения

Проект содержит автоматические тесты для проверки корректности работы административной панели и публичного API.
Тесты написаны с использованием фреймворков Mocha, Chai и библиотеки Supertest.
Они не затрагивают рабочую базу данных – для тестов используется отдельная тестовая база.
Требования

    Node.js >= 18

    npm

    Отдельная база данных PostgreSQL (например, specsintez_test)

Настройка тестового окружения

    Создайте тестовую базу данных
    Выполните в PostgreSQL (через psql или pgAdmin):
    sql

    CREATE DATABASE specsintez_test;

    Задайте параметры подключения к тестовой базе
    В файле .env добавьте строку:
    env

    TEST_DATABASE_URL=postgres://postgres:ваш_пароль@localhost:5432/specsintez_test

    Или задайте отдельные переменные:
    env

    TEST_DB_HOST=localhost
    TEST_DB_PORT=5432
    TEST_DB_NAME=specsintez_test
    TEST_DB_USER=postgres
    TEST_DB_PASSWORD=ваш_пароль

    Убедитесь, что db.js корректно переключается на тестовые настройки при NODE_ENV=test (уже настроено в проекте).

    Установите зависимости (если ещё не установлены):
    bash

    npm install

Запуск тестов

Выполните в корне проекта:
bash

npm test

Скрипт автоматически установит переменную окружения NODE_ENV=test и запустит все тесты из папки test/.
Структура тестов

Все тесты находятся в папке test/. Вспомогательные функции для создания и очистки таблиц — в test/setup.js.
Тесты разбиты по разделам:
Файл	Описание
products.test.js	Создание товара, валидация полей, уникальность slug
news.test.js	Создание, редактирование, удаление новостей; проверки slug
categories.test.js	Создание, редактирование, удаление категорий; защита от удаления родителя с детьми
package-types.test.js	Создание, редактирование, удаление типов упаковок; запрет удаления используемого типа
public-api.test.js	Проверка публичных API‑эндпоинтов: /api/news, /api/categories, /api/products, /api/random-products, /api/products/:id, /api/products-by-slug/:slug
Что проверяется

    Корректность ответов (статус-коды 200, 201, 302, 400, 404).

    Валидация форм – обязательные поля, формат slug, уникальность.

    Бизнес-логика – невозможность удалить родительскую категорию с дочерними, запрет удаления используемого типа упаковки.

    Фильтрация и рекурсивные запросы – в публичных API проверяется фильтрация по категории, возврат только активных записей, работа параметра all=true.

    Структура ответов – наличие полей, типы данных, вложенные объекты (категории, изображения, типы упаковок).

Примечания

    В тестовой среде отключены CSRF‑защита и принудительная аутентификация. Это сделано для упрощения тестирования и не влияет на производственный режим.

    Все изменения в тестовой базе откатываются после каждого тестового сценария, поэтому тесты можно запускать многократно без риска испортить данные.

    При ошибках, связанных с подключением к базе, проверьте TEST_DATABASE_URL и доступность сервера PostgreSQL.
